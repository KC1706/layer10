# Layer10 Considerations Writeup

This document addresses three crucial architectural challenges in designing an organizational memory system, beyond what was explicitly implemented in the prototype codebase. 

---

### 1. How to connect unstructured chat to structured Jira tickets

Connecting fluid, unstructured chat (like Slack/Teams) to structured work systems (like Jira/Linear) involves bridging two fundamentally different semantic spaces: *Intent/Discussion* vs. *Action/Tracking*.

**The Mechanism:**
1. **Chat Ingestion & Signal Detection**: Chat channels emit high-volume noise. We need to route chat events through a classification model that identifies "actionable intents" or "status updates".
2. **Entity Resolution (Cross-Domain)**: The extraction layer must be aware of Jira ticket IDs (e.g., "PROJ-123") and Slack thread IDs. We treat structured tickets simply as another `Entity` type.
3. **Graph Bridging**: 
   When the unstructured chat discusses "the auth bug", the extractor generates a claim:
   `[Entity: chat_thread_45] -> (discusses) -> [Entity: concept_auth_bug]`
   We use our **semantic deduplication engine** (`dedup.py`) to connect this concept to the structured Jira ticket:
   `[Entity: concept_auth_bug] -> (is_tracked_by) -> [Entity: jira_PROJ_123]`
4. **The User Experience**: When viewing the Jira ticket in the visualizer, the KnowledgeRetriever pulls all attached Slack threads seamlessly because the `concept_auth_bug` node acts as the bridge.

---

### 2. Handling Long-Term Memory Drift (Durable vs. Ephemeral)

As organizations change, facts become outdated (drift). "V2 is launching Q3" might change to "V2 is launching Q4." If we simply overwrite, we lose context. If we keep everything, we generate hallucinations.

**The Mechanism:**
1. **Temporal Validity Modeling**: Implemented in Phase 4 of the prototype. Every claim is tagged by the LLM with its `temporal_validity` (e.g., "Until August", "Current as of v1.2").
2. **Decay Functions**: In the `KnowledgeRetriever` (Phase 5 query time), edges in the graph receive a "Recency Discount". If two claims conflict (same Subject/Predicate but different Objects), the system evaluates the timestamp of the backing `Evidence`.
3. **Durable vs. Ephemeral Classification**:
   - *Durable*: Architectural decisions, historical PR merges, post-mortem summaries. These edges decay very slowly.
   - *Ephemeral*: Daily standup updates ("I am blocked on X"). These edges decay rapidly (within days).
   - This distinction is generated by the LLM during Phase 2 Extraction and stored as a metadata weight on the graph edge. 
4. **Garbage Collection (Compaction)**: Ephemeral edges whose temporal validity has definitively expired are moved from `claims` to a `claims_archive` table, significantly speeding up active graph queries while still retaining the capability for historical lookbacks.

---

### 3. Retrieval Permissions & Source Access Limits

A knowledge graph is only as useful as it is secure. An engineer shouldn't use the graph to surface internal HR discussions or private compensation data.

**The Mechanism:**
1. **Evidence-Level Authorization (Zero-Trust Retrieval)**: 
   The graph structure (Nodes) itself is not inherently classified, but the relations (Claims) and the grounding text (Evidence) belong to distinct Source Systems with specific Access Control Lists (ACLs).
2. **The "Context Pack" Gatekeeper**:
   In Phase 5, before `KnowledgeRetriever` yields a `Claim` to the user, it checks the `source_id` associated with the `Evidence`. 
   If the query context is from a user without access to a Slack channel or Jira board, **the claim is redacted**.
3. **Handling Graph Bleed**:
   If an entity `Project_X` is retrieved, but all edges pointing to *why* Project_X is delayed require private access, the Retriever returns `Project_X` as an Entity but sterile of its private predicate links.
4. **Token Injection Architecture**: When a user connects to FastAPI, they supply an OAuth token. The Retriever resolves their ID against an internal ACL map, attaching `user_scopes=[]` to the `QueryRequest`, filtering SQL returns dynamically.
